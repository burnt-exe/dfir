What is DFIR Triage?

Triage is the **rapid assessment phase** of Digital Forensics & Incident Response (DFIR). Its purpose is to determine:

  • What is happening?
  • How severe is it?
  • What should be acted on FIRST?

During security incidents, time is the enemy. The first 5‚Äì15 minutes define whether an attacker is stopped early or allowed to escalate.

Triage bridges the gap between detection and full investigation.

NIST Mapping

NIST SP 800-61 defines triage within the Detect and Respond phases:

  • DE.AE-1 ‚Äî Detection events analyzed
  • DE.AE-3 ‚Äî Event severity determination
  • RS.AN-1 ‚Äî Forensic analysis begins
  • RS.CO-2 ‚Äî Containment actions initiated

NIST SP 800-137: Continuous Monitoring Alignment

Continuous Monitoring feeds triage with:

  • Log telemetry
  • Endpoint events
  • Network detections
  • Threat intel indicators
  • Baseline deviations

MITRE ATT&CK Mapping for Triage

Triage aligns with MITRE ATT&CK during early detection of attacker behaviors:

  • Initial Access ‚Äì Phishing, exploitation
  • Execution ‚Äì Scripts, payloads, PowerShell
  • Persistence ‚Äì Registry keys, scheduled tasks
  • Privilege Escalation ‚Äì Token manipulation, UAC bypass
  • Lateral Movement ‚Äì SMB, RDP, PSRemoting
  • Command & Control ‚Äì DNS tunneling, HTTPS beacons
  • Exfiltration ‚Äì Data staging, cloud sync abuse

Effective triage requires pattern recognition ‚Äî mapping alerts and evidence to ATT&CK tactics quickly.

Types of Triage

1. Host-Based Triage

Performed on a workstation, server, VM, or cloud instance.

  • Running processes
  • Network connections
  • Autoruns / persistence
  • Security logs
  • Malicious binaries
  • Execution history (PSReadline, ShimCache, Amcache)

2. Network Triage

Focuses on malicious traffic indicators such as:

  • Beaconing patterns
  • Abnormal DNS queries
  • Port scanning
  • Lateral movement
  • VPN anomalies
  • Cloud exfiltration routes

3. Cloud Triage

Identity-first environments require cloud-focused triage:

  • IAM role misuse
  • OAuth token abuse
  • Impossible travel
  • Privilege escalation via policy changes
  • API key leakage
  • S3/GCS/Blob over-access

4. Memory (RAM) Triage

Critical when analyzing real-time malware behavior:

  • Injected processes
  • In-memory payloads
  • Encrypted network sessions
  • Volatile attacker artifacts

Evidence Prioritization During Triage

You cannot analyze everything at once. High-value evidence includes:

  • Indicators of Compromise (IOCs) ‚Äî hashes, domains, IPs
  • Process execution events ‚Äî suspicious binaries
  • Authentication logs ‚Äî failed logons, MFA bypass
  • Network connections ‚Äî C2 servers, anomalous ports
  • Cloud audit logs ‚Äî IAM changes, privilege spikes
  • EDR alerts ‚Äî correlation findings

The aim is not to collect everything ‚Äî it is to collect what matters NOW.

Evidence Handling & Acquisition (DFIR/NIST-Compliant)

Effective triage requires disciplined evidence handling. Evidence must be acquired quickly, safely, and in a way that preserves integrity for deeper investigation.

This section outlines best practices from NIST SP 800-61, SWGDE, and forensic standards.

Principles of Evidence Handling

  • Preserve before you analyze ‚Äî snapshot or acquire first, investigate second.
  • Volatile evidence first ‚Äî RAM, running processes, network connections.
  • Follow chain-of-custody ‚Äî who touched what, when, and why.
  • Minimize footprint ‚Äî every command changes the system.
  • Document everything ‚Äî timestamps, tools used, errors, and results.

Volatile Evidence Order (Live Response)

When a system is running, volatile data disappears quickly. This is the preferred acquisition order:

  1. CPU registers, cache
  2. RAM / Memory image
  3. Network connections / active sessions
  4. Running processes
  5. Open files / handles
  6. System time + BIOS time
  7. Logged-in users
  8. Disk data

Chain of Custody (NIST Standard)

Chain of custody must show every human or system that interacted with evidence.

  • Who acquired the evidence
  • When it was acquired (timestamp)
  • What tool or method was used
  • Where it was stored
  • How integrity was validated (e.g., hashes)

DFIR Triage Process (Full Lifecycle)

Triage is not a single action ‚Äî it is a structured, repeatable DFIR lifecycle. This section breaks down the entire triage process in stages, aligning with NIST SP 800-61, CISA MSSP guidelines, and modern SOC workflows.

Triage Levels (Tiered SOC Model)

  • Tier 1 (Alert Intake) ‚Äì Validate, classify, prioritize.
  • Tier 2 (Incident Analyst) ‚Äì Deep triage, correlation, evidence review.
  • Tier 3 (DFIR/Threat Hunter) ‚Äì Forensics, threat intelligence, containment strategy.

Tier 1 analysts perform the majority of triage. Tier 2+ takes over only if severity is high or evidence is complex.

Triage Stage Breakdown

Stage 1 ‚Äî Alert Validation

Confirm the alert is accurate, relevant, and worth escalating.

  • Check SIEM rule logic
  • Compare against baselines
  • Check for similar past alerts
  • Lookup IoCs in threat intel sources

Goal: Determine if this is real or noise.

Stage 2 ‚Äî Identify Affected Assets

Determine what is at risk:

  • Endpoints / servers
  • Cloud identities
  • User accounts
  • Applications
  • Databases
  • Network segments

Goal: Map the attack surface.

Stage 3 ‚Äî Severity Classification

Severity is based on impact + likelihood.

  • Critical ‚Äì Active threat, lateral movement, data loss risk
  • Major ‚Äì Suspicious behavior with possible escalation
  • Minor ‚Äì Low-risk anomaly, monitoring required

Goal: Decide urgency & escalation.

Stage 4 ‚Äî Rapid Investigation

Perform accelerated analysis:

  • Check active processes
  • Identify suspicious network activity
  • Analyze cloud logs
  • Verify user activity
  • Review relevant EDR events

Goal: Confirm threat presence.

Stage 5 ‚Äî Containment Decision

Containment options include:

  • EDR isolation
  • Account disable
  • Firewall block
  • Token/session revocation
  • Quarantine of artifacts

Goal: Stop the threat before it spreads.

Stage 6 ‚Äî Documentation

Proper documentation ensures IR continuity:

  • Timeline of actions
  • Evidence collected
  • Severity rating
  • Containment justification
  • Escalation rationale

Goal: Ensure auditability & chain-of-custody.

Stage 7 ‚Äî Escalation Path

Escalate to one or more groups based on severity:

  • Tier 2 SOC Analyst
  • DFIR Team
  • Threat Hunting Team
  • Cloud Security Team
  • Identity & Access Management (IAM)
  • Incident Commander

Goal: Ensure the right team engages.

Stage 8 ‚Äî Handoff to Incident Response

When triage confirms malicious activity, the incident is fully activated:

  • IR classification assigned
  • Containment & eradication plan built
  • Forensics begins (memory, disk, cloud artifacts)
  • Communications plan initiated
  • Business owners notified

Goal: Move to IR lifecycle smoothly.

Interactive Triage Decision Tree

This interactive decision tree guides analysts during rapid incident assessment.

Step 1 ‚Äî Is the alert confirmed malicious?

If YES ‚Üí proceed to Step 2.
If NO ‚Üí document and close with justification.

Step 2 ‚Äî Is the threat active RIGHT NOW?

If YES ‚Üí isolate affected systems immediately (EDR/SOAR automation recommended).

If NO ‚Üí proceed to impact assessment.

Step 3 ‚Äî What is the asset criticality?
  • Tier 0: Domain controllers, IAM, cloud root accounts
  • Tier 1: Production servers, financial systems
  • Tier 2: Standard employee workstations

Higher criticality = higher triage severity.

Step 4 ‚Äî Is lateral movement observed?

If YES ‚Üí escalate to IR immediately (Critical severity).

If NO ‚Üí continue risk evaluation.

Step 5 ‚Äî Classify Severity

Critical: Active threat, privilege escalation, C2, data access

Major: Suspicious activity with potential escalation

Minor: Low-risk anomaly
